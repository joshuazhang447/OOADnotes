Week 12: Design Principles (Chapter 8)

Design principles are guidelines that help us write flexible, maintainable, and reusable code. These are general rules that increase cohesion, reduce coupling, and make a system easier to modify.

1. Single Responsibility Principle (SRP)

Definition: Every object/class in your system should have a single responsibility.

  • Every class should have one reason to change.
  • It implies one clearly defined responsibility.
  • SRP reduces confusion and promotes high cohesion.

Measuring SRP: LCOM

We can measure adherence to SRP using LCOM (Lack of Cohesion of Methods). This helps quantify SRP violations.

  • Rule of thumb: Methods that don't share common data or purpose belong in different classes.

2. DRY (Don't Repeat Yourself)

Definition: Every piece of knowledge should exist in one and only one place in your code.

  • If you have logic, data, or rules repeated in more than one place, you have created a maintenance trap.
  • When something changes, you have to find and fix it everywhere. If you miss one spot, the system breaks.

Refactoring Checklist

Use this checklist to identify DRY violations:

  1. Do you see similar logic repeated in different methods?
  2. Do multiple classes use the same calculation or rule?
  3. Do you update multiple files when a requirement changes?

Code Example: Violation vs. Refactor

Violating DRY:

public class Greeting {
    public void morningGreeting() {
        System.out.println("Good morning");
    }
    public void eveningGreeting() {
        System.out.println("Good evening");
    }
}

Refactored (DRY Compliant):

public class Greeting {
    public void greet(String time) {
        System.out.println("Good " + time);
    }
}

3. Open-Closed Principle (OCP)

Definition: Software entities should be open for extension, but closed for modification.

  • You should be able to add new behaviors (extend) without rewriting or modifying the existing, working code.
  • Example: Imagine a GuitarSpec class. If we want to add a new instrument type, we shouldn't have to modify the existing class. Instead, if we use an Interface, we can add new types without breaking the rest of the system.

4. Liskov Substitution Principle (LSP)

Definition: If S is a subclass of T, we should be able to use S anywhere T is expected without breaking behavior.

The Classic Example: Square vs. Rectangle

Mathematically, a square is a rectangle. However, in software design:

  • A Square is not behaviorally a Rectangle.
  • Why? Because in a rectangle, you can change the width independently of the height. In a square, changing one side must change the other.
  • If you substitute a Square where a Rectangle is expected, the code might try to change the width only, which violates the rules of a Square (or breaks the logic of the code expecting a Rectangle).

Concept: Delegation

What is Delegation?

  • Delegation is primarily used as an alternative to inheritance for achieving code reuse.
  • It involves handing over a responsibility to another class rather than inheriting it.
  • It promotes more flexible software designs.
