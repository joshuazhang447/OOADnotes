System Modeling & Design Principles

This document summarizes key concepts in Object-Oriented Analysis and Design (OOAD), including the distinction between analysis and design, principles of good design, object modeling, UML diagrams, and core programming concepts.

Analysis vs. Design

These are two fundamental phases in software development that answer different questions.

  • Analysis: Focuses on WHAT the system should do.
    • Its goal is to understand the problem.
    • Involves discovering requirements and defining system behavior.
    • Example: "A system must track instruments in inventory."
  • Design: Focuses on HOW the system will do it.
    • Its goal is to plan the solution.
    • Involves creating a plan to implement requirements in software.
    • Example: "We'll use an `Instrument` class with subclasses like `Guitar` and `Mandolin`."

What Makes a Good Design?

Design is the process of planning how to solve a problem using software. It serves as a blueprint for developers. A good design has several key attributes:

  • Easy to understand: Code is clear and logical.
  • Flexible / Maintainable: Can be easily modified or updated without breaking.
  • Allows for future development: Extensible to accommodate new features.
  • Reusable: Components can be used in other parts of the system or in different projects.
  • Adoptable: Can be integrated with other systems or technologies.

Characteristics of a Design Plan:

  1. Has a purpose: It solves a real-world problem.
  2. Describes context: Explains how something works within the larger system.
  3. Contains enough detail to implement: Provides clear instructions for developers.
  4. Exists at different levels: Can be high-level (architecture) or low-level (class design).

Core Object-Oriented Principles

These are the four pillars that form the foundation of Object-Oriented Programming (OOP).

  • Abstraction: Hide the complexity and show only the essential features of an object.
  • Inheritance: Allow a new class (subclass) to inherit attributes and methods from an existing class (superclass) to promote code reuse and extension.
  • Polymorphism: The ability for an object to take on many forms. A single interface can be used for a general class of actions.
  • Encapsulation: Bundle the data (attributes) and methods that operate on the data into a single unit (class), and restrict access to some of the object's components (i.e., keep it private).

Object Modeling: Static vs. Dynamic Views

Object Modeling is the process of identifying and describing the objects in a system, including their attributes, behaviors, and relationships. It helps answer two main questions:

  1. What are the key objects in a system? (e.g., `Sale`, `Register`, `Item`, `Customer`)
  2. How do they interact to make the system work? (Who sends messages, what actions happen, and when?)

1. Static Model (Focuses on Structure)

  • Diagram Used: Class Diagram
  • Purpose: Describes what classes exist and how they relate to each other. It represents the system's structure.
  • Example Classes:
    • Register: Handles checkout and records the sale.
    • Sale: Represents a single transaction.
    • Item: Represents each product being sold.

2. Dynamic Model (Focuses on Behavior)

  • Diagrams Used: Sequence Diagram, Communication Diagram
  • Purpose: Describes how objects interact with each other over time. It shows the flow of messages and the sequence of events.
  • Example Sequence (Making a Sale):
    1. Cashier enters an item.
    2. Cashier tells the Register to enterItem().
    3. The Register creates a Sale.
    4. The Sale retrieves the Item information.
    5. The Sale updates itself with the new line item.

UML Class Diagrams

What is a UML Class Diagram?

A UML Class Diagram is a visual representation of the static structure of a system.

  • It shows classes, their attributes, operations (methods), and their relationships (associations).
  • It's like a blueprint that shows what things exist in your software and how they are connected.

Why Use a Class Diagram?

  • To understand what objects our system needs.
  • To clarify how data and behavior are organized.
  • To communicate design ideas clearly between analysis and stakeholders.
  • To serve as a bridge between the analysis and design phases.

The Two Perspectives in Class Diagrams

Class diagrams can be viewed from two different perspectives, depending on the phase of development.

  • Conceptual View (Domain Model):
    • Represents real-world concepts.
    • Describes the business or system in plain language, without programming details.
    • Focuses on logical entities and their relationships.
    • Helps in communicating with non-technical users and stakeholders.
  • Design View (Design Class Diagram):
    • Represents the software implementation of the domain model.
    • Adds implementation details like attributes with data types and method signatures.
    • Defines navigability (which objects know about other objects).
    • Helps developers know what to build.

Key Implementation Concepts

Abstract Class vs. Interface

  • Abstract Class:
    • A blueprint for other classes.
    • Can contain shared attributes and methods (with implementation).
    • Defines abstract methods that subclasses must implement.
  • Interface:
    • Defines a set of behaviors (methods) that a class agrees to implement.
    • Does not provide any code or implementation for its methods.
    • Defines what a class can do, not how it does it.

The DRY Principle (Don't Repeat Yourself)

DRY is a fundamental principle aimed at reducing repetition of software patterns and code.

Why is duplicate code bad?

  1. Hard to maintain: A bug or required change must be fixed in multiple places.
  2. Hard to extend: Adding new, similar functionality often means writing the same code again.
  3. Inconsistent: Small variations between the duplicated code blocks can cause logic mismatches and subtle bugs.
