Week 5: Introduction to UML and System Modeling

Analysis vs. Design

Before building a system, we must distinguish between two critical phases: analysis and design. They answer two fundamentally different questions:

  • Analysis ⇒ Do the right thing.
    This phase is about understanding the problem domain and defining what the system needs to do. It focuses on functional requirements and business rules.
  • Design ⇒ Do the thing right.
    This phase is about figuring out how the system will accomplish its goals. It focuses on architecture, technology choices, and the implementation details.

Modeling with UML is essential in both phases to ensure clarity and correctness.

Unified Modelling Language (UML)

UML is a standardized modeling language consisting of an integrated set of diagrams. It helps developers and stakeholders visualize and understand the different aspects of a system. UML diagrams are broadly categorized into two types: Structure Diagrams and Behavioral Diagrams.

Structure Diagrams

These diagrams show the static structure of the system and its parts. They depict the elements that must be present in the system, such as classes, objects, and components.

  1. Composite Structure Diagram
  2. Deployment Diagram
  3. Package Diagram
  4. Profile Diagram
  5. Class Diagram (Shows the classes in a system and their relationships)
  6. Object Diagram
  7. Component Diagram

Behavioral Diagrams

These diagrams illustrate the dynamic behavior of the system. They show how the system responds to events and how its components interact over time.

  1. Activity Diagram
  2. Use Case Diagram (Describes system functionality from a user's perspective)
  3. State Machine Diagram
  4. Interaction Diagrams (A sub-category that includes the following):
    • Sequence Diagram (Shows object interactions in time sequence)
    • Communication Diagram
    • Interaction Overview Diagram
    • Timing Diagram

This lecture will focus primarily on Use Case Diagrams to model system functionality.

Use Case Diagram: Banking System Example

A use case diagram captures the high-level functionality of a system by showing how external entities, called actors, interact with the system's use cases (features).

Task: Create a Use Case Diagram for a Banking System

We need to identify the actors, use cases, and the associations between them for a simple banking system.

1. Identify Actors:

Actors are roles that interact with the system. They can be people or other systems.

  • Customer: The primary user of the bank's services.
  • Bank Employee: A staff member who manages accounts and assists customers.
  • ATM: An automated system that provides certain banking services.

2. Identify Use Cases:

Use cases are the specific goals or tasks that actors want to achieve with the system.

  • Withdraw Cash
  • Deposit Funds
  • Check Balance
  • Transfer Funds
  • Open Account
  • Close Account

3. Define Associations:

Associations are the lines that connect actors to the use cases they can perform.

Related Concepts: CRUD Operations

Many system operations can be broken down into four fundamental functions known as CRUD. This acronym is essential in designing data-driven applications.

  • C - Create: Add new data to the system (e.g., creating a new bank account).
  • R - Retrieve / Read: Read or view existing data (e.g., checking an account balance).
  • U - Update: Modify existing data (e.g., changing a customer's address).
  • D - Delete: Remove data from the system (e.g., closing an account).

When analyzing a system like a Point of Sale (POS), a use case like "Manage Item" would involve CRUD operations on the item's data (ItemID, Name, Price, etc.).
