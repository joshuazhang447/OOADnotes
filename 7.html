Requirements Analysis & Domain Modeling

This document covers the initial phases of Object-Oriented Analysis, focusing on how to capture system requirements using Use Cases and User Stories, and then translating them into a conceptual Domain Model using techniques like CRC Cards and textual analysis.

Defining Requirements: Use Cases vs. User Stories

Both Use Cases and User Stories help define what a system should do, but they approach it from different levels of detail and formality.

Use Cases

A Use Case is a detailed description of how a user (or "actor") interacts with the system to achieve a specific goal. It describes the system's behavior from the user's perspective.

  • Actor: Who or what interacts with the system (e.g., a person, another system).
  • Main Flow: The "happy path" or success scenario.
  • Preconditions: What must be true before the use case starts.
  • Postconditions: What must be true after the use case successfully ends.
  • Alternatives/Exceptions: What happens if something goes wrong or an alternative path is taken.

Example: Dog Door System

  • Use Case Name: Open Dog Door
  • Actors: Dog (primary), System, Bark Recognizer
  • Precondition: The dog door is closed.
  • Main Flow:
    1. The dog barks.
    2. The Bark Recognizer detects the bark.
    3. The System opens the door for 5 seconds.
  • Postcondition: The door closes after 5 seconds.
  • Alternative: The bark is not recognized.

User Stories

A User Story is a short, simple description of a feature told from the perspective of the person who desires the new capability, usually a user or customer of the system. It's often described as "a promise for a conversation."

  • Characteristics: Small, Independent, Testable (S.I.T.).
  • Benefits: Lightweight, flexible, focuses on user needs, and easy to modify. Commonly used in Agile/XP methodologies.
  • Format: As a [user type], I want [goal] so that [benefit].

Example: Dog Door System

"As a [dog owner], I want [the door to open when my dog barks] so that [my dog can go outside safely]."

Building a Domain Model

A Domain Model represents concepts, attributes, and relationships in the real-world problem area. It's important to remember that a domain model is not yet a software design; it's a visual dictionary of the key abstractions in the domain.

Steps to Build a Domain Model:

  1. Find Conceptual Classes: Identify key nouns from use cases and requirements.
  2. Draw UML Diagrams: Represent the classes and their connections visually.
  3. Show Relationships: Define associations between classes.
  4. Add Attributes and Multiplicity: Detail the properties of each class and the cardinality of their relationships.

Identifying Conceptual Classes

A common technique for finding classes is Textual Analysis.

  • Identify all nouns in the use cases or requirements documents. These are potential classes.
  • Eliminate duplicates and irrelevant nouns.
  • Keep a list of potential classes, even if you are unsure. They can be refined later.

Object Collaboration and CRC Cards

What is Object Collaboration?

Object collaboration is the "teamwork" between objects. Since one object can't do everything by itself, it must work with other objects to complete a task. This is achieved by objects sending messages to each other (i.e., calling each other's methods).

CRC Cards (Class-Responsibility-Collaborator)

CRC cards are an informal, team-based technique to define how objects collaborate. Each card represents a single class in the system and is divided into three sections:

  • Class Name: The name of the object or concept.
  • Responsibilities: What the class knows or does (its behaviors).
  • Collaborators: Other classes it works with to fulfill its responsibilities.

Example: Dog Door CRC Card

Class: Dog Door
Responsibilities Collaborators
- Open & Close the door - Bark Recognizer
- Remote

Multiplicity in UML

Multiplicity (or cardinality) in UML shows how many instances of one class can be associated with one instance of another class. It describes the rules of a relationship.

Common Multiplicity Notations

Notation Meaning Example
1 or 1..1 Exactly one A Dog Door is connected to exactly one Bark Recognizer.
0..1 Zero or one (optional) A Dog may have a Collar (but not always).
* or 0..* Zero or many A Dog can have many Barks.
1..* One or many An Owner has at least one Dog.
