Advanced Design & Analysis Concepts

This document covers advanced design principles such as cohesion, scalability, and iterative development, along with a deeper look into analysis techniques and class design.

Iterative Development

A core concept in modern software development is that design is not a one-time event.

  • A good design isn't done once—it evolves.
  • Each development iteration improves upon the existing design.
  • Software design is a cycle, not a straight line.

Commonality vs. Variability Analysis

This analysis technique helps identify what parts of a system are common and what parts vary. This is crucial for creating flexible and reusable designs, often leading to the use of abstract classes and interfaces.

Simple Analysis Example: What do a Turkey and a Canada Goose have in common?

  • They both have feathers, are birds, and can fly or swim. These are their commonalities.

Instrument Example:

  • Commonality (What do all instruments share?):
    • Manufacturer
    • Type
    • Price
    • Serial Number
    • Specifications
  • Variability (What differs between instruments?):
    • Style
    • Number of strings
    • Wood type
    • Shape
    • Tuning

Advanced Design Principles

These principles guide developers in creating software that is robust, flexible, and easy to manage over time.

Cohesion

Cohesion refers to how focused a class is. A class should do one thing and do it really well. A key sign of good cohesion is that a class has only one reason to change.

  • High Cohesion (Good): Each class handles its own distinct purpose, making the system more flexible.
    • Example: A CustomerAccount class manages only balance and transactions.
  • Low Cohesion (Bad): One class tries to do too much, making it rigid and hard to change.
    • Example: A generic Utility class that handles file I/O, sending emails, and image processing.

Scalability

A system is scalable if it can easily grow or handle more features, users, or data without needing major changes to the code. In other words, it can expand smoothly and handle growth easily.

Maintainability

A system is maintainable if it is easy to understand, fix, and modify in the future without breaking other parts of the code.

Abstract vs. Concrete Classes

Classes can be broadly categorized as either abstract or concrete, each serving a different purpose in object-oriented design.

Abstract Class

  • Acts as a blueprint for other classes.
  • Cannot be used to create actual objects directly.
  • Represents a general concept that is shared by more specific types.
  • May contain abstract methods (methods without code) that subclasses must implement.

Concrete Class

  • A complete class.
  • You can create objects directly from it.
  • It provides actual implementations for all of its methods.

Advantages of Using Abstract Classes

  • Clean and Logical: Separates general ideas from specific ones. Common properties and methods go in the abstract parent class.
  • Reusability: Designed to be extended, promoting code reuse.
  • Safe: It prevents developers from creating meaningless objects (e.g., creating an "Instrument" object instead of a specific "Guitar" or "Piano").

Data Structures: Map (or Dictionary)

What is a Map?

A Map (often called a Dictionary in other languages) is a collection of key-value pairs.

  • Each key is unique within the map.
  • Each value is the data associated with that key.

A real-life dictionary is a perfect analogy:

  • The word is the key.
  • The definition is the value.

Benefits of Using Maps

  1. Flexibility: Can store various types of data.
  2. Reusability: A general-purpose structure for lookups.
  3. Scalability: Efficient at handling large amounts of data for lookups.
  4. Generic Searching: Provides a fast and standard way to find data by its key.
