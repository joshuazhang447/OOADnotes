Week 13: Domain Models & Design Patterns

This week focuses on conceptualizing the system through Domain Models, understanding the importance of Design Patterns, and structuring applications using the Model-View-Controller (MVC) concept.

1. The Domain Model

What is a Domain Model?
A Domain Model is a picture (diagram) or description that shows:

  • The real-world concepts in the system.
  • Their attributes (data they store).
  • Their relationships (how they connect).

It represents the problem we are trying to solve. Importantly, a Domain Model is:

  • NOT the UI.
  • NOT the Database.
  • NOT the Code.

Example: A POS (Point of Sale) System has a domain model that includes: Sale, Product, Payment, Cashier, Store.

2. Design Patterns

What is a Design Pattern?

A Design Pattern is a named solution to a common problem in software design.

  • It is something that has been proven to work again and again.
  • It is a reusable idea we can apply in different situations.
  • It acts like a template, recipe, or blueprint.

Why do we name them?

Naming creates a shared vocabulary among developers. If one developer says "Use a Factory here," every other developer understands the architectural implication quickly.

Common Design Patterns

  • Singleton: Ensures one instance only exists.
  • Strategy: Allows choosing behavior at runtime.
  • Observer: Notifies multiple objects automatically (events).
  • Factory: Creates objects without showing how they are created.
Example: The Singleton Pattern

Problem: Want only one instance of something in the whole system.

Real World: There is only one principal in the school. There is only one president of the company.

In Software: One printer manager, one database connection manager, or one game engine instance.

3. System Architecture (Separation of Concerns)

Domain Class vs. UI Class

To keep the system clean and reusable, we separate logic from the interface.

  • Domain Class (Business Logic):
    • Stores real business rules.
    • Performs calculations.
    • Knows data & logic.
  • UI Class (Presentation):
    • Only handles user interaction.
    • Only shows output.
    • Passes requests to the domain class.

Why not connect Domain Objects directly to UI?

Because the UI changes many times in many forms (Web, Mobile, Desktop, Touchscreen), but the business logic should remain the same.

The MVC Analogy (Restaurant)

We can explain the relationship between Model, View, and Controller using a restaurant analogy:

  • Model (The Chef): Represents the business logic.
    • Not allowed to take orders (no user interaction).
    • Focuses on the "cooking" (data/behavior).
  • View (The Waiter): Represents the User Interface.
    • Not allowed to cook (no logic).
    • Handles user interaction.
  • Controller (Order System): The bridge.
    • They communicate through the controller.
