Week 11: Design Principles & Analysis

This week covers iterative development, core design principles like cohesion and scalability, commonality analysis, class structures, and data structures (Maps).

Iterative Development

Design is not a static process; it is a cycle.

  • Good designs aren't done once—it evolves.
  • Each iteration improves the design.
  • Software design is a cycle, not a straight line.

Design Principles (Design II)

Key characteristics of a robust software architecture include Cohesion, Scalability, and Maintainability.

1. Cohesion

Cohesion defines how focused a class is. A class should do one thing really well and have one reason to change.

  • High Cohesion (Good): Each class handles its own purpose (flexible).
    • Example: A Customer Account class manages only balance and transactions.
  • Low Cohesion (Bad): One class tries to do too much. This makes it hard to change.
    • Example: A Utility class that handles mixing, emailing, painting, etc.

2. Scalable

A system is scalable if it can easily grow or handle more features or data without needing major changes to the code.

  • In other words: it expands smoothly and can handle growth easily.

3. Maintainable

A system is maintainable if it is easy to understand, fix, and modify in the future without breaking other parts of the code.

Commonality vs. Variability Analysis

This analysis technique is used to distinguish between general concepts and specific implementation details.

Basic Analysis Example

Question: What do a Turkey and a Canada Goose have in common?

  • Answer: Feathers, fly, swim, birds.

Instrument Analysis Example

We separate what all instruments share from how they differ.

  • Commonality (What do all instruments share?):
    • Manufacturer
    • Type
    • Price
    • Serial #
    • Specifications
  • Variability (What differs?):
    • Style
    • Number of strings
    • Wood type
    • Shape
    • Tuning

Classes: Abstract vs. Concrete

Understanding the relationship between blueprints and actual objects.

Abstract Classes

  • Are like a blueprint.
  • Cannot be used to create actual objects (can't build from it directly).
  • usually represents a general concept that is shared by more specific types.
  • May contain abstract methods (methods without code yet) that subclasses must implement.

Concrete Classes

  • A complete class.
  • It can create objects from it.
  • It provides actual implementations for all its methods.

Advantages of Using Abstract Classes

  • Clean and Logical: Separates general ideas from specific ones. Common properties and methods go in the abstract parent.
  • Reusability: Designed to be extended.
  • Safe: It prevents creating meaningless objects (e.g., preventing the instantiation of a generic "Shape" when you need a "Circle").

Data Structures: Map (or Dictionary)

Definition

A Map in Java is a Collection of key-value pairs.

  • Each key is unique.
  • Each value is the data associated with that key.

Real Life Analogy

If we take a dictionary:

  • The Word is the Key.
  • The Definition is the Value.

Benefits of Using Maps

  1. Flexibility
  2. Reusability
  3. Scalability
  4. Generic Searching
