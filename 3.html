Week 3 Notes: Requirements Gathering & FURPS Model

Requirement Gathering

Requirement gathering is a foundational aspect of software development. It is the process of identifying, analyzing, and documenting the complete requirements for a system.

This process is based on input from various sources. The primary providers of requirements are:

  • Users (the people who will interact with the system)
  • Stakeholders (individuals or groups with an interest in the project's success)
  • Clients (the entity commissioning and funding the project)

Ultimately, requirement gathering defines two critical aspects of the system to be developed:

  • What a system must do.
  • How it should behave based on input from users, stakeholders and clients.

Categorizing Requirements: The FURPS Model

To ensure all aspects of a system are considered, requirements are often categorized. A widely used model for this is FURPS, which is an acronym for five key categories of software quality attributes.

  • F - Functional: Refers to the core features, capabilities, and security of the system.
    • Features
    • Capabilities
    • Security
  • U - Usability: Relates to the user experience and ease of use.
    • Human factors (e.g., aesthetics, consistency)
    • Help & Documentation
  • R - Reliability: Concerns the system's ability to perform consistently and recover from failures.
    • Frequency of failure
    • Recoverability
    • Predictability
  • P - Performance: Defines the system's responsiveness, speed, and efficiency.
    • Response times
    • Throughput
    • Accuracy
    • Availability
  • S - Supportability: Involves the ease of maintaining, adapting, and configuring the system after deployment.
    • Adaptability
    • Maintainability
    • Internationalization
    • Configurability
